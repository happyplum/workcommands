description: Restructure bloated or overlapping project memories without losing durable knowledge.
---

You are executing the `/restructure-memory` command.

Treat `$ARGUMENTS` as the memory topic, memory names, or the specific restructuring goal.

Operate as a command-first memory-governance workflow. Do not stop at “invoke a memory skill”; drive the restructuring to a finished, verified result.

Required behavior:

1. Resolve the restructuring scope from `$ARGUMENTS`, the current task, or the current project memory inventory.
2. Start from a full relevant memory inventory, not intuition.
3. Prefer the safe restructuring ladder:
   - split
   - trim
   - rename
   - preserve
   - merge
   - rewrite
   - delete
4. Keep durable experience. Do not delete memories merely because they are old, overlapping, or inconvenient.
5. When a memory is replaced, first create or update the target structure, then delete only after verifying the useful knowledge survived.
6. If restructuring changes repository truth carriers such as README or related governance docs, sync them in the same command path.
7. End with a concrete restructuring report: what was kept, split, renamed, rewritten, merged, or deleted, and why.

If `$ARGUMENTS` is empty and no stable topic is obvious, first audit the current project's reusable memories and then proceed with the clearest restructuring candidate.

Use the existing memory-governance skills as reference material when useful, but the command itself owns the operational flow and completion standard.
