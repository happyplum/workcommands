# Sisyphus / Omo / Docs 清理

## 概述

将 `.sisyphus` / `.omo` / `docs/` 清理视为**验证优先的执行产物清理**，而非盲目文件清除。核心规则：**探测目录 → 盘点 → 分类内容 → 验证状态 → 去重持久化知识 → 删除临时产物 → 证明清理结果**。

## 目录探测

新版本 omo 已将工作目录从 `.sisyphus` 迁移到 `.omo`；同时，项目根目录的 `docs/` 可能混放用户文档与 Agent 执行产物。清理时需要把三者作为**并行候选目标**处理：

1. **探测存在性**：检查项目中 `.omo`、`.sisyphus` 和 `docs/` 是否存在
2. **确定工作区目标**：
   - 仅 `.omo` 存在 → 清理 `.omo`
   - 仅 `.sisyphus` 存在 → 以 `.omo` 为迁移目标进行清理（见下方迁移协议）
   - **两者都存在** → 先合并 `.sisyphus` 中的内容到 `.omo`，再清理两者（见下方迁移协议）
   - 两者都不存在 → 不设置 `$WORKSPACE`，但若 `docs/` 存在仍必须继续执行 docs 清理
3. **确定 docs 目标**：
   - `docs/` 存在 → 将 `docs/` 加入清理盘点范围
   - `docs/` 不存在 → 跳过 docs 清理
4. **结束条件**：仅当 `.omo`、`.sisyphus` 和 `docs/` 都不存在时，才报告无需清理并结束
5. 后续所有步骤中，将 `.omo` / `.sisyphus` 的目标目录简称为 `$WORKSPACE`；`docs/` 始终以 `docs/` 单独称呼

## 迁移协议

### 何时触发

- `.sisyphus` 和 `.omo` 同时存在 → 必须触发
- 仅 `.sisyphus` 存在，且盘点后发现**存在无法删除的活跃内容**（如未完成计划、活跃检查点） → 必须触发，将活跃内容迁移到 `.omo`

### 迁移流程

1. **盘点两者**：
   - 仅 `.omo` 存在时 → 盘点 `.omo`
   - 仅 `.sisyphus` 存在时 → 盘点 `.sisyphus`，识别活跃与可删除条目
   - 两者共存时 → 分别列出完整目录树，交叉对比

2. **分类 `.sisyphus` 内容**：
   - **可安全删除的临时产物**（已完成计划、过期证据等） → 直接删除
   - **活跃/无法确认状态的内容**（未完成计划、活跃检查点、`boulder.json` 中仍指向的工作） → **迁移到 `.omo` 对应位置**
   - **持久化知识** → 按记忆处理协议处理，不迁移文件

3. **合并策略**（两者共存时）：
   - `.omo` 中已有的内容 → 以 `.omo` 版本为准，忽略 `.sisyphus` 中的同名文件
   - `.sisyphus` 独有的有价值内容 → 迁移到 `.omo` 对应位置
   - `.sisyphus` 独有的临时/过期内容 → 直接删除

4. **完成迁移后**：
   - 对 `.omo` 按正常流程执行清理
   - 删除 `.sisyphus` 中所有已迁移或已清理的内容
   - 若 `.sisyphus` 变空，移除 `.sisyphus` 根目录

## Docs 目录清理

### 概述

项目中的 `docs/` 目录常被 Agent 用于存放执行产物（计划文件、技术债追踪、架构分析等），与面向用户的持久文档混放。清理目标是：**删除已验证的 Agent 执行产物，保留所有面向用户的文档，在删除前提取有价值的持久化知识**。

### 探测条件

1. **探测存在性**：检查项目中是否存在 `docs/` 目录
2. **不存在** → 跳过 docs 清理，仅处理 `.omo` / `.sisyphus`
3. **存在** → 进入 docs 清理流程

### 内容分类

对 `docs/` 中每个文件/子目录进行分类：

#### 应保留（面向用户或维护者的持久文档）

- **帮助/使用文档**：用户手册、操作指南、FAQ、快速入门等
- **API/接口文档**：API 参考、接口规范、SDK 文档等开发者文档
- **项目说明/历史**：项目架构概述、CHANGELOG、版本历史、roadmap、设计决策记录（ADR）等
- **历史规划/项目记录**：已明确作为项目历史、产品演进或架构决策背景保留的规划文档
- **仍活跃的维护文档**：当前未解决且仍指导维护工作的技术债、风险清单、兼容性约束等
- **判断依据**：内容是面向项目使用者或维护者的持久参考，不因单个任务完成而过期；即使文件名包含 plan/debt/design，也不能仅凭文件名删除

#### 应清理（Agent 执行产物）

- **已完成计划文件**：实现计划、迁移计划、阶段计划、执行清单等，且相关工作已在代码/配置/文档现实中验证完成
- **已废弃计划文件**：路线已被替代、目标已取消，且不再作为项目历史或决策背景保留
- **已解决/过时技术债追踪**：TECH_DEBT.md、debt 清单、风险清单中记录的问题已修复、迁移到 issue tracker，或与当前代码现实不再相符
- **临时设计草案/分析**：架构分析、方案对比、临时调研文档、Agent 生成的方案评审文档，且长期价值已沉淀到 README、ADR、记忆或代码注释
- **判断依据**：内容是 Agent 为完成特定任务而产出的临时文档，任务完成后价值已沉淀到代码、记忆或其他持久载体；删除不会削弱用户帮助、API 参考、项目说明或历史规划的可用性

#### 未知

- 无法明确分类 → 保留并标记待后续处理

### Docs 清理流程

1. **盘点**：列出 `docs/` 完整目录树，读取每个文件的内容概要
2. **分类**：按上述标准将每个条目分为「应保留」/「应清理」/「未知」
3. **验证状态**：
   - 对「应清理」条目：验证相关工作是否确实已完成（检查代码现实、记忆、issue tracker）
   - 对计划文件：确认计划项对应的代码、配置、测试、文档或发布现实是否已完成/废弃/迁移
   - 对技术债追踪文档：逐条检查问题状态；未解决且仍准确的条目应保留或归并到维护文档，已解决/失效的条目标记可清理
   - 对设计草案/分析：确认其结论是否已进入 README、ADR、架构文档、代码现实或记忆
   - 不确定时归类为「未知」
4. **持久化知识提取**：删除前，对有长期价值的 Agent 产物执行记忆处理协议（同 `.omo` 清理）；若内容更适合用户或维护者直接阅读，应优先沉淀到 README、ADR 或项目说明文档，而不是只写入记忆
5. **删除已验证的执行产物**：仅在步骤 3-4 完成后删除
6. **证明结果**：列出 `docs/` 剩余内容，报告清理前后对比，并说明哪些用户文档、API 文档、项目说明或历史规划被保留

### Docs 删除检查清单

仅在以下**全部**为真时才删除 docs 中的文件：

- 内容已确认为 Agent 执行产物，非面向用户的持久文档
- 相关工作在代码/仓库中已验证完成（或已废弃/已迁移）
- 任何独特持久化知识已提取到记忆或确认为冗余
- 删除不损害项目使用者的文档可用性
- 清理后状态可立即验证

不确定时，保留并标记待后续处理。

## 加载条件

以下任一条件为真时加载：

- 用户要求清理、修剪、归档、重置或移除 `.sisyphus`、`.omo` 或 `docs/` 内容
- 目录包含可能已过期的计划、便签、证据、状态文件或空文件夹
- `docs/` 中混放了 Agent 执行产物和用户文档，需要区分清理
- 用户希望仅保留有用的架构笔记和用户文档，同时删除已完成的执行产物
- 清理决策取决于条目是否已完成、活跃、已在记忆中重复或可安全删除

## 强制规则

1. 从完整盘点开始。禁止仅凭文件名分类。
2. 在删除计划、便签或 `docs/` 文档前，先在代码/仓库现实中验证完成状态。
3. 将 `boulder.json`、计划文件、证据、笔记、技术债追踪、设计草案视为**有状态载体**，直到证明可废弃。
4. 仅在检查已有记忆确认无重叠后，才保留持久化架构知识；若知识面向用户或维护者阅读，优先落到 README/ADR/项目文档。
5. 禁止将任务日志镜像到记忆。仅保留持久化的 `guardrails`（防回归）、`reference`（参考）、`topology`（拓扑）、`contract`（契约）知识。
6. 仅在唯一价值被证伪或已保存到其他位置后，才删除临时产物。
7. 仅在确认真正为空后才移除空目录。
8. 以证据结束：展示剩余目录状态或验证目录已不存在。

## 分类检查

**验证后**对每个条目分类：

- **活跃状态** — 当前计划/会话状态、活跃检查点、使用中的产物、仍准确的未解决技术债 → 保留
- **用户/维护者文档** — 帮助文档、API 文档、项目说明、历史规划、ADR、版本历史 → 保留
- **已完成临时产物** — 已结束的计划、证据输出、执行日志、过期追踪器、已解决技术债清单、临时设计分析 → 删除
- **持久化知识** — 架构、防回归经验、拓扑、可复用规则 → 通过已有/更新记忆、README、ADR 或项目文档保留
- **未知** — 无法证明状态或用户价值 → 保留至验证完成

## 执行顺序

1. **探测所有目标目录**：按上方「目录探测」流程确定 `$WORKSPACE`（`.omo` 或 `.sisyphus`）；同时探测 `docs/` 是否存在。若两者共存，或仅 `.sisyphus` 存在且有活跃内容无法删除，则执行「迁移协议」

2. **盘点所有目标**：分别列出 `$WORKSPACE` 和 `docs/`（若存在）的完整目录树；识别计划、便签、证据、状态文件、空目录、Agent 执行产物

3. **验证状态，而非推断**：
   - 对 `$WORKSPACE` 内容：确认工作在代码/仓库中确实已完成；检查 `boulder.json`（或等效状态文件）是否仍指向活跃工作；即使用户说「已完成」，仍需验证
   - 对 `docs/` 内容：确认 Agent 执行产物相关工作是否确实已完成；技术债问题是否已解决；即使用户说「已完成」，仍需验证

4. **删除前审查持久化知识**：先读取已有记忆；若持久化知识已存在，禁止重复；若笔记或文档包含独特长期价值，压缩为聚焦的记忆更新；使用 `/restructure-memory` 原则：一条记忆 = 一个主要职责

5. **仅删除已验证的临时产物**：仅在步骤 3-4 完成后删除：
   - `$WORKSPACE` 中的已完成计划、过期便签、证据输出、可废弃状态文件
   - `docs/` 中的已验证 Agent 执行产物
   - 确认空后移除空目录；若 `$WORKSPACE` 变空，一并移除根目录

6. **证明结果**：列出所有目标目录的剩余内容，或验证目录已不存在；报告已删除项、已保留项、为何无持久化知识丢失

## 删除检查清单

仅在以下**全部**为真时才删除：

- 状态已验证，非假定
- 不是当前活跃执行状态的来源
- 不是面向用户或维护者的持久文档
- 任何独特持久化知识已保留或证明确属冗余
- 移除减少噪音而不产生第二事实来源
- 清理后状态可立即验证

不确定时，保留并标记待后续处理。

## 记忆处理协议

保留笔记衍生知识前的顺序：`list_memories` → `read_memory` 检查重叠 → 决定（保留已有/更新已有/新建/不操作）→ 执行 → 然后才删除来源

**禁止存储**：会话 ID、一次性执行日志、临时审计笔记、验证输出文本、大量复制的计划历史

## 报告结构

完成时报告：删除前已验证 → 已删除 → 持久化知识已保留/有意不保留 → 最终证据（`剩余目录树` 或 `路径不存在`）

**边界**：使用 `/restructure-memory` 处理跨多条记忆的存储级重组；使用 `/cleanup-sisyphus` 处理 `.sisyphus` / `.omo` 执行产物清理和 `docs/` Agent 产物清理。当清理发现持久化知识缺口时，使用 `/restructure-memory` 原则进行保留步骤。
